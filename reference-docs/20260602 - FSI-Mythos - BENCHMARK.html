💡 해석

두 도구 모두 오탐 0(Precision 1.0)이지만, Semgrep 기본 규칙은 Python os.system 명령주입과 C strcpy 버퍼오버플로우를 탐지하지 못했습니다(전용 규칙셋 필요). FSI-Mythos는 의미론적 추론으로 데이터 흐름(사용자 입력 → 위험 싱크)을 추적해 탐지했습니다. SAST는 Precision·속도, LLM 에이전트는 Recall·범용성에 강점 — 상호 보완적입니다.

⚠️ 공정성 주의

Semgrep은 --config auto 기본 규칙만 사용했습니다. p/security-audit·p/owasp-top-ten 등 전용 규칙셋이나 Pro 규칙을 쓰면 Recall이 올라갑니다. 또한 본 코퍼스는 소규모라 절대 수치보다 방향성으로 해석해야 합니다.

Section 08

Tier-2 재현성 (pass@k)

LLM 에이전트는 비결정적이므로 동일 케이스를 여러 번 독립 호출해 탐지가 일관적인지 측정해야 합니다. 대표 6케이스(명령주입·약한암호·인증우회의 취약/안전 쌍)를 각 3회 독립 호출(end-to-end)하여 측정했습니다.

측정 지표

  • detect_rate: 3회 중 탐지(validated≥1)된 비율 (취약은 1.0, 안전은 0.0이 이상적)
  • pass@k: k회 시도 시 1회 이상 탐지될 확률 (관측 조합 기반)
  • CWE Jaccard: 회차별 탐지 CWE 집합의 일관성 (1.0=완전 동일)
  • flaky: 회차마다 탐지 여부가 흔들리는 케이스

✅ 재현성 결과

비결정적 LLM(temperature=1.0)임에도 탐지 판정은 완전히 안정적입니다. 취약 3케이스는 3/3 일관 탐지, 안전 3케이스는 3/3 정상 통과, flaky 케이스 0개. pass@1과 pass@3이 모두 1.0이라 단 1회 스캔으로도 신뢰할 수 있음을 의미합니다.

📊 CWE 일관성 관찰

명령주입·인증우회는 회차별 CWE 집합이 완전 동일(Jaccard 1.0). weak-crypto는 0.733으로, 핵심 CWE(327 약한해시·329 고정IV·798 하드코딩키)는 3회 모두 일관 탐지하되 부가 CWE만 회차별로 달랐습니다(run1-2: CWE-20 입력검증, run3: CWE-353 무결성). 즉 핵심 취약점 판정은 안정적이고, 추가 맥락 CWE에서만 약간의 다양성이 나타납니다.

🎯 성능 인사이트

취약점이 없는 코드는 Hunter가 빈 결과를 내 후속 단계를 건너뛰어 7배 빠릅니다(19s vs 140s). 실무에서 대부분 코드가 정상임을 고려하면 평균 비용은 낮아집니다. pass@k를 높이면 재현성↑·시간/비용↑.
